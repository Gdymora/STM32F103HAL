ELF (Executable and Linkable Format) — це контейнер який містить кілька речей одночасно:

- машинний код (скомпільований з C, не асемблер але схоже)
- символи — імена функцій і змінних з адресами (gpio_init, main і т.д.)
- секції — .text, .data, .bss як в лінкер скрипті
- debug інформація — звідки який рядок C коду, для GDB

Коли робиш objdump -d hal_blink.elf — він дизасемблює машинний код назад в асемблер щоб ти міг читати. Але сам ELF зберігає саме байти машинного коду а не текст асемблера.

Тобто ланцюжок такий:

main.c          ← твій C код
  ↓ компілятор
main.o          ← об'єктний файл (машинний код + символи)
  ↓ лінкер
hal_blink.elf   ← фінальний ELF (код + debug + символи + адреси)
  ↓ objcopy
hal_blink.bin   ← чисті байти без нічого зайвого → це і пишемо в Flash
.bin це те що реально заливається в чіп. .elf потрібен для GDB щоб знав де яка функція і міг показувати рядки C коду.

1 Так, objdump -d — це один з найкорисніших інструментів для розуміння що реально відбувається.

Спробуй прямо зараз без grep — просто:

arm-none-eabi-objdump -d hal_blink.elf | less
Побачиш щось таке:

08000168 <gpio_init>:
 8000168:   b580        push    {r7, lr}
 800016a:   b086        sub     sp, #24
 800016c:   af00        add     r7, sp, #0
 800016e:   6078        str     r0, [r7, #4]
Зліва направо:
- 08000168 — адреса в Flash
- b580 — реальні байти які записані в чіп
- push {r7, lr} — що ці байти означають в асемблері

І можна навіть порахувати — arm-none-eabi-size hal_blink.elf показував text=376. Якщо порахувати байти в objdump — вийде рівно 376.

Ще корисна команда — побачити всі символи з адресами:

arm-none-eabi-nm hal_blink.elf
Виведе список всіх функцій і змінних з їх адресами в пам'яті. Зручно коли хочеш знайти де саме в Flash живе конкретна функція.

2 А що можна на асемблері писати
Так, можна писати прямо в C через asm:

void delay_asm(uint32_t n) {
    __asm volatile (
        "1: subs %0, %0, #1 \n"  // n--
        "   bne  1b          \n"  // якщо не нуль — назад
        : "+r" (n)
    );
}
Це точніша затримка ніж while(n--) — рівно 2 інструкції на ітерацію, незалежно від -O0 чи -O2.

Але це так — для розуміння. Що реально цікавого можна робити з objdump і суміжними інструментами:

Подивитись розмір кожної функції:
arm-none-eabi-objdump -d hal_blink.elf | grep "^[0-9a-f]* <"
**Порівняти код з -O0 і -O2** — видно як компілятор оптимізує, іноді результат шокуЗнайти звідки береться bloatat** — чому прошивка раптом стала більша:
arm-none-eabi-nm --size-sort hal_blink.elf
**Перевірити що -nostdlib реально не тягне зайвого:**
arm-none-eabi-readelf -S hal_blink.elf

Покаже всі секції — якщо немає нічого крім .text, .data, .bss — все чисто.

Спробуй скомпілюй з -O2 замість -O0 в Makefile і порівняй розмір і дизасемблер — різниця буде помітна.
