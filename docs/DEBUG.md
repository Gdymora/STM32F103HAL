Корисні команди:

```bash
arm-none-eabi-objdump -d hal_blink.elf | grep -A 20 "gpio_init"
```
 що там відбувається з `port = 0x40010800`:

```bash
arm-none-eabi-objdump -d hal_blink.elf | grep -A 20 "rcc_enable"
```

Ось як правильно "заглянути в мозок" твоему процесору, щоб побачити ту саму адресу в таблиці векторів.

### Крок 1: Запусти сервер зв'язку
В одному терміналі (залиш його відкритим):
```bash
st-util
```
Він має написати `Listening at *:4242`.

### Крок 2: Запусти GDB для ARM
У другому терміналі (де твій файл `button_exti.elf`):
```bash
gdb-multiarch button_exti.elf
```

### Крок 3: Команди всередині GDB
Тепер введи ці команди по черзі:

```gdb
# 1. Підключаємось до заліза
target extended-remote :4242

# 2. Скидаємо процесор (опціонально)
monitor reset halt

# 3. Читаємо адресу переривання EXTI1 (позиція 23)
x/1xw 0x0800005C
```

---

### Що ми шукаємо?

Коли ти введеш `x/1xw 0x0800005C`, GDB покаже щось на кшталт:
`0x0800005c: 0x08000145`

**Дивись на останню цифру (молодший нібл):**
1.  **Якщо там непарне число (закінчується на 1, 3, 5, 7, 9, b, d, f):**
    З таблицею векторів усе гаразд! Процесор знає адресу і знає, що це Thumb-режим. Проблема десь у логіці `EXTI_PR` або `NVIC`.
2.  **Якщо там парне число (закінчується на 0, 2, 4, 6, 8, a, c, e):**
    **БІНГО!** Процесор намагається стрибнути в режим ARM (якого в нього немає), падає в `HardFault` і зависає. Це значить, що лінкер або стартап неправильно позначили функцію як код.
3.  **Якщо там `0x00000000` або `0xffffffff`:**
    Таблиця векторів зміщена. Процесор шукає переривання "в пустоті".

### Як перевірити саму функцію?
Введи в GDB:
```gdb
p EXTI1_IRQHandler
```
GDB напише адресу функції. Порівняй її з тим, що ти побачив за адресою `0x0800005C`. Вони мають збігатися (адреса в таблиці може бути на 1 більша за адресу функції — це нормально для Thumb).
