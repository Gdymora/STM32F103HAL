**Що таке U-Boot**

U-Boot — це маленька програма-завантажувач, яка вмикається відразу після подачі живлення на плату. Її задача — підготувати залізо (DDR, тактування, периферію), знайти ядро Linux на flash, завантажити його в RAM і передати йому керування разом зі списком параметрів.

Аналогія: BIOS/UEFI на ПК. Той самий рівень — між "залізо щойно ввімкнулось" і "ОС запускається".

На відміну від BIOS, U-Boot — open source, написаний на C, і має повноцінну командну оболонку. Можна перервати завантаження кнопкою Ctrl+C і отримати prompt `=>` де можна виконувати команди: дивитись пам'ять, читати flash, міняти параметри, навіть завантажити kernel з мережі через TFTP.

**Що ми будемо з ним робити (у порядку складності)**

1. **Підключитись через UART і просто побачити його** — Ctrl+C під час boot, prompt `=>`, базові команди `help`, `printenv`, `version`, `mmc info`. Це сама діагностика — нічого не змінюємо, просто дивимось.

2. **Розібрати реальні параметри завантаження** — що таке `bootcmd`, `bootargs`, env-партиція. Зрозуміти кожне слово в тому рядку який передається в kernel (`console=ttyFIQ0 root=/dev/ubiblock0_0 ...`).

3. **Змінити налаштування в рантаймі** — `setenv bootdelay 10`, `saveenv`. Це показує як працює U-Boot environment без жодної перекомпіляції.

4. **Перекомпілювати власний U-Boot** — `./build.sh uboot` в SDK. Перший раз — без жодних змін, просто щоб упевнитись що toolchain працює і можемо отримати власний `uboot.img`.

5. **Написати патч** — модифікуємо джерела U-Boot. Додаємо власний banner ("Custom UBoot by Alex"), збільшуємо bootdelay у defconfig, додаємо власну команду `mystatus`, міняємо autoboot prompt. Зберігаємо все як `git diff`.

Чудова картинка, тепер все стало кристально ясно — і ось важлива поправка до того що я писав:

**На основних гребінцях є UART2 — піни 1 і 2!**

Дивись на ліву сторону:
- **Pin 1: GPIO1_B2_d / UART2_TX_M1**
- **Pin 2: GPIO1_B3_u / UART2_RX_M1**
- **Pin 3: GND**

Тобто UART2 (debug console для U-Boot) виведений **на основний жовтий гребінець, на самих перших пінах згори**. Це **функція M1** — той самий debug UART який нам треба, той самий `0xff4c0000`.

Це **набагато простіше** ніж паяти на TX0/RX0/GND біля USB-C — там pads дрібні, а тут можна просто встромити дроти в гребінець або причепити dupont-кабель.

Що означає для статті: Розділ 3 ("Підключення до debug UART з паяльником") треба переписати. Замість паяння — просто 3 dupont-провідники на пін 1, 2, 3. Це навіть краще для статті — паяння відлякує початківців, а dupont-кабель будь-хто впорається.

Якщо ти захочеш паяти — можна додати окремо як альтернативу для тих хто хоче чисте рішення без проводів що стирчать. Але дефолтний шлях у статті — гребінець + dupont.

**По плану — повністю погоджуюсь з твоєю пропозицією.**

Робимо так:
1. Йдемо крок за кроком на реальному залізі
2. Кожен крок — нотатки в чат: команди, виводи, що зламалось
3. Я веду паралельно конспект-довідник (markdown, як `uart_ubuntu_guide.md` і `br2_package_guide.md` що в тебе вже є)
4. Помилки і дивні речі обов'язково фіксуємо — це найцінніше для статті
5. Коли весь маршрут пройдено — повертаємось до docx, перепишемо з реальними дампами замість моїх шаблонних


**Перший крок — підключення UART і ловля U-Boot prompt.** Потрібно:

1. Dupont-провідники (мама-мама або мама-тато залежно від того що в тебе для CH340)
2. CH340 USB-TTL (той самий що для STM32)
3. minicom (вже є з Місяця 1)

Підключення:
```
Luckfox pin 1 (UART2_TX_M1)  →  CH340 RX
Luckfox pin 2 (UART2_RX_M1)  →  CH340 TX
Luckfox pin 3 (GND)          →  CH340 GND
```

CH340 в один USB-порт ПК, Luckfox через USB-C в інший USB-порт. Не з'єднувати VCC адаптера з нічим — Luckfox живиться сам.

Запускаєш:
```bash
minicom -b 1500000 -D /dev/ttyUSB0
```

Перевір що ttyUSB0 — це правильний пристрій (`ls /dev/ttyUSB*` після підключення). Якщо CH340 — буде `ttyUSB0`, якщо CP2102 — теж `ttyUSB0`, якщо FTDI — теж.

Натиснути **RESET** на платі. Давай нотатки що побачив — починаючи від першого байта і до prompt `=>`. Якщо взагалі нічого не з'являється — теж пиши, розберемося разом (швидше за все 90% випадків — переплутані TX/RX, це норм).

Окей, типова ситуація. 90% випадків — щось з підключенням або налаштуваннями minicom. Йдемо діагностувати по черзі.

**Крок 1: перевір що minicom не в локальному echo режимі і що hardware flow control вимкнено**

У minicom натисни `Ctrl+A` потім `O` не нуль (configure). Зайди в **Serial port setup**. Перевір:

```
A - Serial Device      : /dev/ttyUSB0
E - Bps/Par/Bits       : 1500000 8N1
F - Hardware Flow Control : No        ← КРИТИЧНО, має бути No
G - Software Flow Control : No
```

Якщо `Hardware Flow Control` = Yes — це 99% твоя проблема. CH340 має тільки TX/RX, без RTS/CTS. Якщо minicom чекає CTS — він не покаже жодного байта.

Натисни `F` щоб перемкнути на No, потім Esc, потім **Save setup as dfl** (щоб не повторювати кожен раз), потім Exit.

**Крок 2: переконайся що baud rate саме 1500000**

У `E - Bps/Par/Bits` має бути `1500000 8N1`. Не 115200, не 9600. Якщо minicom не приймає 1500000 — то старі версії могли не вміти таких швидкостей. Перевір версію:

```bash
minicom --version
```

2.8 (як у тебе) — вміє точно.

**Крок 3: потисни RESET ще раз і одночасно дивись у minicom**

Якщо все одно тиша — швидше за все TX/RX переплутано. Це найчастіша помилка. Поміняй місцями два дроти на CH340 (TX↔RX), залиш GND як був. Знов RESET.

**Крок 4: швидкий sanity-check самого CH340**

У окремому терміналі (не закриваючи minicom — або закрий тимчасово):

```bash
sudo dmesg | tail -20
```

Має бути щось типу `ch341-uart converter now attached to ttyUSB0`. Якщо нема — проблема з самим адаптером, не з налаштуваннями.

Ще можеш закоротити TX і RX на самому CH340 (просто з'єднати дві ніжки пінцетом або провідником), зайти в minicom і щось набрати — букви мають з'являтись на екрані (loopback). Це доводить що сам адаптер живий.

**Крок 5: чи правильні піни**

Подивись на плату ще раз. Pin 1 — це найперший пін у верхньому лівому кутку (з боку USB-C, найближчий до нього). Лічить треба з тієї сторони де USB-C. На картинці яку ти прислав, цифра "1" стоїть біля UART2_TX_M1 — це і є pin 1.

Перевір ще раз:
- Pin 1 → CH340 RX
- Pin 2 → CH340 TX  
- Pin 3 → GND CH340

Дивно. Hardware Flow Control тепер No, але loopback все ще не працює. Перевіримо зовнішнім інструментом, бо в minicom щось ще може бути не так.

**Закрий minicom повністю:** `Ctrl+A` → `X` → `Yes`.

**Перевір що TX і RX на CH340 справді закорочені.** Покажи фото CH340 з перемичкою якщо можеш — щоб я бачив які саме піни ти з'єднав. На більшості CH340-адаптерів є піни типу `VCC GND TXD RXD 3V3 5V` або щось подібне. Закорочувати треба саме `TXD` (TX) і `RXD` (RX).

**Тест без minicom — через python:**

```bash
sudo apt install python3-serial -y
```

Потім:

```bash
python3 -c "
import serial, time
s = serial.Serial('/dev/ttyUSB0', 1500000, timeout=1)
s.write(b'HELLO\n')
time.sleep(0.2)
print('Got:', s.read(100))
s.close()
"
```

З підключеною перемичкою TX-RX на CH340 повинно вивести:
```
Got: b'HELLO\n'
```

Якщо вивело `Got: b''` (порожньо) — або перемичка не контактує, або адаптер несправний.

Якщо вивело `HELLO` — адаптер живий, проблема саме в minicom.

Варіант 3 — впевнитись що ttyUSB0 справді відкривається:
```bash
python3 -c "
import serial
s = serial.Serial('/dev/ttyUSB0', 9600, timeout=1)
print('Port opened:', s.is_open)
print('Settings:', s.get_settings())
s.close()
"
```
Якщо вивід нормальний — порт відкривається, проблема саме в фізичній петлі.

Хм, `\x00\x00\x00\x00...` — це нулі. Дані йдуть, але це не текст boot-логу. Кілька можливих причин:

1. **Швидкість не та** — найімовірніше. Якщо швидкість сильно не співпадає, в потоці часто пробігають саме нулі або 0xff. На 1500000 ми бачимо 0x00 — спробуй **115200**.

2. **Інверсна полярність** — рідше, але буває.

**Перевір на 115200:**

```bash
python3 -c "
import serial
s = serial.Serial('/dev/ttyUSB0', 115200, timeout=2)
print('Listening on 115200... press RESET on Luckfox now')
import sys
while True:
    data = s.read(256)
    if data:
        sys.stdout.write(data.decode('ascii', errors='replace'))
        sys.stdout.flush()
"
```

Або через minicom: `Ctrl+A` → `O` → Serial port setup → `E` → `C` для 115200 → Enter → Esc.

Натисни RESET на платі.

Якщо побачиш текст — вітаю, ми в U-Boot territory! Якщо знов нулі — спробуй ще одну швидкість, **1500000 baud** офіційна, але деякі CH340 з нею просто не справляються (пам'ятаєш я в Частині 3 написав "deякі CH340 не тримають 1500000" — і ось тобі практичне підтвердження).

**Ще варіанти швидкостей які варто спробувати по черзі:**
- 115200 — найімовірніше зараз
- 1500000 — офіційна для раннього boot
- 1152000 — буває якщо CH340 не вміє 1500000 точно
- 921600 — теж зустрічається
- 57600 — рідше, але буває

Працює на 115200

🎯 **Прекрасно! зловив U-Boot prompt!**:

```
Hit key to stop autoboot('CTRL+C'):  0
=> <INTERRUPT>
=> <INTERRUPT>
```

Це означає що **в U-Boot prompt**! Той `<INTERRUPT>` після `=>` — це U-Boot реагує на твої безперервні Ctrl+C (кожен Ctrl+C перериває команду яку він зараз виконує). Тепер просто **відпусти Ctrl+C** і у тебе буде чистий prompt.

І ще цінні знахідки з дампу:

```
SPL Hotkey: ctrl+c
...
Hotkey: ctrl+c
```

U-Boot **сам каже** який у нього hotkey. Тобто Ctrl+C — правильний.
